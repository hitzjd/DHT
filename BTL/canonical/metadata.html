Metadata for Audiovisual Content

Metadata is specified using an XML document using the XHTML vocabulary. The class attribute is used for all metadata markup. Nested elements are used for grouping, and the optional syntax <abbr title="actual value in prescribed format">ignored text for illustrative purposes</abbr> is used to include text values formatted for display and automatic extraction. Values of type text may occur multiple times with different explicit or inherited values of xml:lang to specify translations; they must not have multiple values in a single instance separated by commas or other delimiters.

File Names

All metadata files must have names ending in one of .xml, .xhtml, .html, .xht, or .htm (case distinctions are ignored.) All such files will be scanned for metadata, and URIs referenced by these files will be resolved relative to the location of these files at the time they are discovered. This behavior can be overridden using the xml:base attribute on XML elements in the metadata file. Note that the case of filenames referred to in URI references must match their actual case, including the case of the file extension. All file and folder names must be unambiguous when converted to lower case, and must consist entirely of printable US-ASCII letters, digits, hyphens, underscores, and full stop/period (U+002E, U+002D, U+0030 - U+0039, U+0041 - U+005A, U+005F, and U+0061 - U+007A in Unicode notation.) There must be exactly one full-stop/period in the filename, and it must appear between the non-empty base filename and the non-empty file extension. No folder should contain more than 1000 files. Files violating these constraints may be ignored or processed incorrectly.

Recommendation for File Names

It is our recommendation that you organize files as follows:

  • Create a top-level folder named according to the identifier (user/company name) assigned to you by the retailer converted to lower case, for example foo-corp.
  • Create a second-level folder for each set of files you will send us. Name it based on the estimated date of delivery, in the form yyyymmdd, e.g. 20060601 for a release on June 1st, 2006.
  • Choose an identifier scheme to consistently refer to each of your products. We recommend ISAN, but GTIN (including EAN, UPC, or ISBN-13 bar-code numbers), or even a company-internal product identifier scheme would work, so long as the identifier is stable over time and file format changes and unambiguously identifies exactly one of your products. Remove all spaces, identifier-independant prefices and punctuation and convert the identifier to upper case. Use this as the third-level folder name.
  • Each file inside the third-level directory has an lower-case base name consisting of a lower-case copy of the third-level folder name optionally followed by an underscore and a disambiguating suffix chosen by you to avoid collision; we recommend no such suffix for the first file, then incrementing decimal digits starting at 2 for subsequent filtes. Each file has a lower-case file extension separated from the base name with a full-stop/period. The extension should be the most commonly used extension for files of the appropriate type.
  • Optional Rule: Prefix each level of folder name with intermediate folder names that are prefices of that name, starting with the shortest prefix that divides the potential name space into 256 or fewer sub-spaces and choosing successively longer prefices by the same algorithm and using the part of the prefix not in common with the previous prefix as the folder name, until there are 256 or fewer identifiers in the sub-space at which point the actial third-level identifier is present as a folder name. ISAN check characters and GTIN check digits are not considered to divide the potential name space, since they are calculated based on preceding characters.
